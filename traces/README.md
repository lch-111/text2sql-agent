# traces/ — 错题本与经验沉淀

> 每次 Self-Correction 修正成功或失败后，
> Agent 自动在此目录创建 Trace 文件。
> 这些记录是 Agent 的"错题本"，
> 用于后续相似场景的快速定位和知识复用。

---

## 目录规则

### 文件命名

```
YYYY-MM-DD_简短描述.md
```

示例：`2025-06-09_province-mapping-error.md`

### 文件结构

每个 Trace 文件遵循 `.harness/traces/trace-template.md` 模板，包含：

1. **触发场景**：用户原始输入、系统状态、复现步骤
2. **原始报错**：完整的错误日志/堆栈
3. **根本原因分析**：直接原因 + 深层原因 + 受影响组件
4. **最终解决方案**：代码修改/术语库更新/Prompt 调整
5. **预防措施**：是否添加了测试/文档/安全规则

---

## 分类

| 类型 | 说明 | 频率 |
|------|------|------|
| 术语映射 | 用户口语无法匹配数据库实际值 | 较高 |
| 字段不存 | 模型生成不存在的字段名 | 中等 |
| 语法错误 | 引号使用错误、JOIN 条件缺失 | 较低 |
| 业务口径 | 聚合时未按业务口径过滤 | 需重视 |

---

## 知识复用

生成 SQL 前，Agent 应检索 `traces/` 目录下是否存在与当前查询相似的场景复盘，
将相关经验作为 Few-shot 示例注入 Prompt。

---

## 定期清理

遵循 neat-freak 规则，每 30 天清理一次：
- 保留所有包含"业务口径变更"的 Trace
- 合并重复类型的 Trace
- 删除超过 90 天的低价值记录

## 相似度去重

新增 Trace 时自动进行相似度检查（基于编辑距离/子串匹配）：
- 若新 Trace 的问题与已有 Trace 相似度 > 0.7，自动合并到现有记录（更新时间和补充信息）
- 成功范例库（如有）同样应用此规则，仅保留有代表性的样本，避免无限膨胀
- 合并策略：保留更详细的那条，合并两条的解决方案
